# Stomp Claw 2.0 Design

## Overview

Complete overhaul of the stomp-claw voice assistant daemon. The proof-of-concept (v1) validated the MIDI pedal control, NeMo transcription, OpenClaw LLM integration, and browser-based viewer. v2.0 rebuilds with a proper architecture for reliability, performance, and a modern UI.

## Core Problems Solved

- **Streaming reliability** — v1 had fragile SSE with padding hacks and timeout-based completion detection. v2 uses WebSockets with explicit lifecycle events.
- **Message persistence** — v1 used flat JSON files with no transactional guarantees. v2 uses SQLite with ACID transactions.
- **Reconnection** — v1 lost state on browser refresh during a stream. v2 persists streaming state to DB; reconnecting clients catch up seamlessly.
- **Monolithic codebase** — v1 was 2300 lines in one file. v2 splits into focused modules.
- **UI** — v1 had embedded vanilla JS with polling. v2 uses TanStack Start with real-time WebSocket updates.

## Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Language | Rust | Already working, performant, single binary |
| Web framework | Axum | Async-first, native WebSocket, tokio ecosystem |
| Real-time transport | WebSocket | Bidirectional, no SSE padding hacks, enables UI text input |
| Database | SQLite | ACID guarantees, zero-config, single-file, embedded |
| Internal architecture | Event bus (tokio::broadcast) | Decoupled modules, multiple consumers per event |
| Project structure | Single crate, module split | Simple for solo project, no workspace overhead |
| Frontend | TanStack Start | Modern React meta-framework, file-based routing, SSR-capable |
| Frontend styling | Tailwind CSS v4 | Utility-first, dark terminal aesthetic |
| Frontend embedding | Separate dev, embedded release | HMR in dev, rust-embed for single binary in release |
| Visual style | Dark terminal/hacker | Near-black bg, monospace, terminal green accents |

## Event System

The event bus is the heart of v2. All state changes flow through a `tokio::broadcast` channel.

```rust
enum Event {
    // MIDI
    PedalDown,
    PedalUp,

    // Recording & Transcription
    RecordingStarted { session_id: String },
    PartialTranscript { session_id: String, text: String },
    RecordingCancelled { session_id: String },
    RecordingComplete { session_id: String, samples: Vec<f32> },

    // Transcription
    FinalTranscript { session_id: String, text: String },

    // Commands (detected from transcript)
    VoiceCommand { command: Command },

    // LLM
    LlmThinking { session_id: String, turn_id: i64 },
    LlmToken { session_id: String, turn_id: i64, token: String, accumulated: String },
    LlmDone { session_id: String, turn_id: i64, full_response: String },
    LlmError { session_id: String, turn_id: i64, error: String },

    // Session
    SessionSwitched { session_id: String },
    SessionCreated { session: Session },
    SessionRenamed { session_id: String, name: String },
    SessionDeleted { session_id: String },

    // Config
    VoiceToggled { enabled: bool },

    // UI-originated
    UserTextMessage { session_id: String, text: String },
}
```

## SQLite Schema

```sql
CREATE TABLE sessions (
    id TEXT PRIMARY KEY,
    name TEXT NOT NULL,
    created_at TEXT NOT NULL,
    last_used TEXT NOT NULL
);

CREATE TABLE turns (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    session_id TEXT NOT NULL REFERENCES sessions(id),
    role TEXT NOT NULL,           -- 'user' or 'assistant'
    content TEXT NOT NULL,
    status TEXT NOT NULL,         -- 'pending', 'streaming', 'complete', 'error'
    created_at TEXT NOT NULL,
    completed_at TEXT,
    FOREIGN KEY (session_id) REFERENCES sessions(id) ON DELETE CASCADE
);

CREATE TABLE config (
    key TEXT PRIMARY KEY,
    value TEXT NOT NULL
);

CREATE INDEX idx_turns_session ON turns(session_id, id);
```

Key design:
- Each user message and assistant response is its own row (not paired)
- `status` on assistant turns tracks streaming state for reconnection
- `content` updated incrementally during streaming (debounced to avoid DB thrash)
- `ON DELETE CASCADE` ensures session deletion cleans up turns

## Module Architecture

```
src/
  main.rs          -- startup, module wiring, tokio runtime
  events.rs        -- Event enum, Command enum, broadcast setup
  midi.rs          -- MIDI listener, pedal state, CC 85 handling
  audio.rs         -- cpal capture, sample buffer, WAV writing
  transcription.rs -- NeMo client (partial + final), cancel detection
  llm.rs           -- OpenClaw client, SSE parsing, streaming
  commands.rs      -- voice command parsing, fuzzy matching, codenames
  db.rs            -- SQLite schema, migrations, CRUD for sessions/turns
  server.rs        -- Axum router, REST endpoints, WebSocket handler
  config.rs        -- config loading/saving, voice toggle state
  beep.rs          -- audio feedback (paplay wrapper)
```

### Module Event Flow

```
              +------- Event Bus (tokio::broadcast) -------+
              |                                             |
    Producers |                                   Consumers |
              |                                             |
    midi.rs   |-> PedalDown/Up                              |
    audio.rs  |-> RecordingStarted/Complete                 |
    trans.rs  |-> PartialTranscript/Final        db.rs    <-|-- persists turns, sessions
    llm.rs    |-> LlmThinking/Token/Done         server.rs<-|-- forwards to WebSocket
    commands  |-> VoiceCommand/SessionX          beep.rs  <-|-- plays audio feedback
    server.rs |-> UserTextMessage/SessionX        llm.rs  <-|-- triggers on FinalTranscript
              |                                   trans.rs<-|-- triggers on RecordingComplete
              +---------------------------------------------+
```

### Startup Sequence

1. Initialize tracing/logging
2. Open SQLite database, run migrations
3. Create `tokio::broadcast::channel(256)`
4. Load config from DB
5. Spawn modules as tokio tasks (each gets tx clone + rx subscriber)
6. Wait for shutdown signal

### Key Change from v1

No more `Arc<Mutex<>>` for shared state. Each module maintains its own local state updated from the event bus. Config, session state, and busy flags all flow as events.

## Streaming Pipeline

```
OpenClaw SSE -> llm.rs -> Event Bus -> +-> db.rs (persist accumulated)
                                       +-> server.rs -> WebSocket -> UI
                                       +-> beep.rs (notify on done)
```

### DB Persistence During Streaming

- `LlmThinking` -> INSERT turn with `status='streaming'`, empty content
- `LlmToken` -> UPDATE content (debounced: every 10 tokens or 500ms)
- `LlmDone` -> UPDATE `content=full, status='complete', completed_at=now()`
- `LlmError` -> UPDATE `status='error'` with accumulated content preserved

### Client Reconnection

On WebSocket connect:
1. Server sends session list with active session
2. Server sends all turns for active session (from SQLite)
3. If stream in progress, latest turn has `status='streaming'` with accumulated content
4. Client renders existing content, then receives remaining `LlmToken` events live

Result: browser refresh, tab crash, multiple tabs — all handled seamlessly.

### Timeout Strategy

```rust
loop {
    let chunk = tokio::time::timeout(
        if first_token { Duration::from_secs(120) }  // thinking phase
        else { Duration::from_secs(10) },              // between tokens
        stream.next()
    ).await;

    match chunk {
        Ok(Some(Ok(bytes))) => { /* parse SSE, publish events */ }
        Ok(Some(Err(e))) => { /* publish LlmError, break */ }
        Ok(None) => { /* stream ended cleanly, break */ }
        Err(_) => { /* timeout fallback, publish LlmDone, break */ }
    }
    first_token = false;
}
```

Timeout is a safety net, not the primary completion mechanism. `[DONE]` and stream-end are detected first.

## REST API

Minimal — only for initial page load. All mutations go through WebSocket.

```
GET  /api/sessions                -- list all sessions
GET  /api/sessions/:id/turns      -- turns for a session (?after=N for pagination)
GET  /api/config                  -- current config
GET  /ws                          -- WebSocket upgrade
GET  /                            -- serve frontend
GET  /assets/*                    -- serve frontend assets
```

### Deployment

- Development: TanStack Start dev server with HMR, proxied to Axum backend
- Release: `npm run build` produces static assets, embedded in Rust binary via `rust-embed`
- Single binary deployment preserved

## Features: Must-Have for 2.0

1. MIDI pedal control (record on hold, process on release)
2. Partial transcription while recording (live preview)
3. Multi-session management via voice commands
4. Voice toggle (on/off) with TTS
5. Text input from UI (secondary to voice)
6. Cancel recording keywords
7. Session management from UI (create/switch/rename/delete)
8. Audio feedback beeps
9. TTS output via ~/bin/speak

## Future Enhancements (Post-2.0)

- Message editing/regeneration from the UI
- Search across conversations

## Migration from v1

- Existing sessions.json and turn JSON files will be migrated to SQLite on first v2 launch
- Session IDs and codenames preserved
- Config.toml migrated to SQLite config table
- v1 code remains on master branch; v2 developed on `v2` branch
